What is Cloud Computing?

Cloud computing is the on-demand delivery of IT resources (computing services) across the internet with pay-as-you-go pricing

You only pay for what you use

Cloud services also expand the traditional IT offerings to include things like Internet of Things (IoT), machine learning (ML), and artificial intelligence (AI)

Because cloud computing uses the internet to deliver these services, it doesnâ€™t have to be constrained by physical infrastructure the same way that a traditional datacenter is



Describing the shared responsibility model


The shared responsibility model defines what entity carries responsibilities under cloud computing

The customer and cloud provider have individual and shared responsibilities


CUSTOMER responsibilities:

The information and data stored in the cloud

Devices that are allowed to connect to your cloud (cell phones, computers, and so on)

The accounts and identities of the people, services, and devices within your organization


CLOUD PROVIDER responsibilities:

The physical datacenter

The physical network

The physical hosts


Responsibilities based on SERVICE MODEL:

Operating systems

Network controls

Applications

Identity and infrastructure



Define Cloud Models

There are THREE main types of cloud models:

Private

Public

Hybrid


PRIVATE clouds are used by a single entity, provide greater control for the company / IT dept with the tradeoff being greater costs and fever benefits of a public cloud

can be hosted in a datacenter either on premise, offsite or even by a third party


PUBLIC clouds are built, controlled and maintained by a third-party cloud provider, and allow for anyone to purchase their services


HYBRID clouds are environments that use both public and private clouds that are interconnected

hybrid clouds allow for a private cloud to surge in increased temporary demand by deploying public cloud resources

hybrid clouds also provide an extra layer of security, and also offer the flexibility to choose what service to keep private / public



a fourth type of cloud is a MULTI CLOUD, which involves multiple cloud providers

this is common for migrating from one cloud provider to another, or using different services from multiple cloud providers


Azure ARC is a set of technologies that helps manage your cloud environment, be it in Azure's public cloud, private cloud, hybrid cloud or multi cloud

Azure also has integration with VMWare - Azure VMware Solution lets customer run their VM workloads in Azure with seamless integration and scalability



Describing the consumption-based model

There are 2 types of expenses to consider when comparing IT ingrastructure models:

CapEx (capital expenditure) vs OpEx (operational expenditure)

OpEx expenses is spending money on services or products over time, like leasing a vehicle, signing up for cloud services, renting office space, etc

Benefits of high availability and scalability in the cloud  

Back to Top

When building or deploying a cloud application, two of the biggest considerations are UPTIME (or availability) and the ABILITY TO HANDLE DEMAND (or scale)

High availability focuses on ensuring maximum availability, regardless of disruptions or events that may occur

Uptime guarantees depend on the service being used

Guarantees are defined by SLAs (service level agreements)

Azure SLAs are represented as a percentage, related to the service or application's availability aka UPTIME

Azure SLAs define downtime and what potential credits can be issued if uptime is not met

Azure SLA percentages range from 99%, 99.9%, 99.95 and 99.99% depending on the service

99% SLA means the downtime is 1.6 hrs/week or 7.2 hrs/mth

99.99 SLA means the downtime is 10 minutes a week or 43.2 minutes/mth

each Azure service has its own SLA


Scalability has 2 types - Vertical and Horizontal

Scalability itself is the ability to adjust resources to meet demand, so the customer does not overpay

Vertical scaling refers to increasing or decreasing the capability of resources (more / less processing power like RAM, CPUs, etc)

Horizontal scaling refers to adding or subtracting resources - aka scaling in (fewer resources needed) or scaling out (more resources needed)



Reliability and Predicatbility in the Cloud

Back to Top

Reliability is the ability of a system to recover from failures and continue to function - one of the pillars of the Microsoft Azure Well-Architected Framework

The cloud is decentralized, so reasources can be deployed in regions all over the world

This means that if one region goes down for whatever reason, another region can pick up the slack


Predictability is another part of the Well-Architected Framework

predictability can be focused on performance predictability or cost predictability


Performance predictability focuses on predicting the resources needed to deliver a positive experience for the customer

Autoscaling, load balancing, and high availability are just some of the cloud concepts that support performance predictability


Cost predictability focuses on predicting or forecasting the cost of the cloud spend with tools like Total Cost of Ownership (TCO) or Pricing Calculator

resources can be tracked in real time to find patters and trends to help plan better resource deployments

Cloud Service Types   Back to Top



Infrastructure as a service (IaaS)

most flexible category of cloud services , as it provides to the customer the maximum amount of control for their cloud resources

In IaaS, the cloud provider is responsible for maintaining the hardware, network connectivity (to the internet), and physical security

The customer is responsible for everything else - all things OS installation, maintenance and configuration, network configuration, database storage

IaaS is essentially renting the cloud datacenter hardware to do what a customer decides to do with it - IaaS places the largest share of responsibility with the customer




Platform as a Service (PaaS)

middle ground / split responsibility between cloud provider and customer, between renting space in a datacenter (IaaS) and paying for a complete and deployed solution (SaaS)

cloud provider is responsible for maintaining the physical infrastructure and its access to the internet, just like in IaaS, and also maintains the operating systems, databases, development tools and business intelligence services

the PaaS model is akin to an employee using a domain-joined machine, in that the IT dept maintains the device updates, patches, etc

well suited to provide a complete development environment without the headache of maintaining all the development infrastructure




Software as a Service (SaaS)

places the most responsibility with the cloud provider and the least responsibility with the customer

least flexible, but the easiest to get up and running / requires the least amount of technical knowledge

customers are responsible for end devices that connect to the system, data put into the system, and user access - nearly eveything else falls on the cloud provider

renting / using a cloud application, like Microsoft 365

Describe Azure physical infrastructure

The core architectural components of Azure may be broken down into two main groupings: the physical infrastructure, and the management infrastructure


Physcial Infra

physical infrastructure for Azure starts with datacenters

datacenters are grouped into Regions and Availability Zones


Regions

a regions is a geographical area on the planet that contains at least one, but potentially multiple datacenters that are nearby and networked together with a low-latency network

when resources are deployed, often-times a region will need to be chosen

some services are only available in certain regions, while others are not region-based, like Microsoft Entra ID, Azure Traffic Manager, and Azure DNS


Availability Zones

Availability zones are physically separate datacenters within an Azure region

each Region has at least 3 separate availability zones to ensure resiliency - but not all Azure Regions support availability zones


Availability Zones are primarily for:

VMs

managed disks

load balancers

SQL databases

Azure Services that support availability zones fall into 3 categories:

Zonal services: customer / cloud engineer pins the resource to a specific zone (for example, VMs, managed disks, IP addresses)

Zone-redundant services: the platform replicates automatically across zones (for example, zone-redundant storage, SQL Database)

Non-regional services: services are always available from Azure geographies and are resilient to zone-wide outages as well as region-wide outages


Region Pairs

Most Azure regions are paired with another region within the same geography (such as US, Europe, or Asia) at least 300 miles away

this allows for the replication of resources across a geography that helps reduce the likelihood of interruptions, regardless of what causes them

Not all Azure services automatically replicate data or automatically fall back from a failed region to cross-replicate to another enabled region - In these scenarios, recovery and replication must be configured by the customer / cloud engineer


Add'l Advantages of Region Pairs:

If an extensive Azure outage occurs, one region out of every pair is prioritized to make sure at least one is restored as quickly as possible for applications hosted in that region pair

Planned Azure updates are rolled out to paired regions one region at a time to minimize downtime and risk of application outage

Data continues to reside within the same geography as its pair (except for Brazil South) for tax- and law-enforcement jurisdiction purposes



Sovereign Regions

Sovereign regions are instances of Azure that are isolated from the main instance of Azure

this can be for legal / compliance reasons



Describe Azure management infrastructure

Management infrastructure includes Azure resources and resource groups, subscriptions, and accounts.


there are FOUR "tiers" of management infrastructure:

Management Groups (MGs)

Subscriptions

Resource Groups

Resources




Resources and Resource Groups

a resource is anything created, provisioned or deployed in Azure

a resource group is a grouping of resources

when a resource is created, the customer / cloud engineer is required to place it into a resource group

a single resource can only be in one resource group at a time

resources in resource groups inherit actions




Subscriptions

using Azure requires a subscription, which provides authenticated and authorized access to Azure products, services, and allows for resource provision

an account can have multiple subscriptions but only is required to have one

multi-subscription accounts can be used to configure different billing models and apply different access-management policies

subscriptions can have TWO TYPES of boundaries to have around products, services and resources:

billing bounadaries - determines how an Azure account is billed for using Azure

access control boundaries - determines or defines specificity of subscriptions




Management Groups

Management Groups are at the top of the "Chain" of management infrastructure

resources -> resource groups -> subscriptions -> Management Groups


subscriptions can be put into containers called management groups and apply governance conditions to the management groups

All subscriptions within a management group automatically inherit the conditions applied to the management group

10,000 management groups can be supported in a single directory

Each management group and subscription can support only one parent

A management group tree can support up to six levels of depth - this limit doesn't include the root level or the subscription level
